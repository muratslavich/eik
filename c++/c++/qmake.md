# qmake

[~~Edit~~](http://marxi.co/#/?provider=evernote\_int\&guid=b7de4aca-bd3d-48ca-87e8-50607655f523\&notebook=) # qmake

[Источник](http://blog.mgsxx.com/?page\_id=1294)

**qmake** просто создает makefile для конкретного компилятора (MSVC, MinGW, …)

_Makefile_ создается на основании значений ряда переменны, которые определяются в .pro файле.

## Переменные

Объявляются неявно с помощью присваивания значений.

**Значения** - это список значенией разделенный пробелами (если значения содержит пробелы то его нужно поместить в “”)

```
1.VAR = value1 value2 # комментарий - от решетки до конца строки
2.VAR = value1 "value with spaces" ..
3...
4.PREFIX = lib
5.VAR1 = value1 value2 
6.VAR2 = v3 \$$VAR1 $${PREFIX}v4  # v3 value1 value2 libv4 \здесь нет
7.# {} нужны в том случае, если имя переменной без них
8.# будет неправильно определяться.
9.# $$PREFIXv4 молча даст пустую строку, т.к.
10.# если переменная не была определена ранее,
11.# ее значение полагается пустым.
```

*
* { } для правильного определения переменной
* список значений должен быть в одну строку
* можно заскейпить символ переноса строки через \\
* через \ можно заскейпить и другие спец символы (\ ” \[ ] { } ( ) $)
* нельзя заскейпить символ #, для нее нужно использовать макрос LITERAL\_HASH

**очистка переменной**

```
1.VAR =      # присвоить пустое значение
2.clear(VAR) # то же самое
3.unset(VAR) # вообще удаляет пременную, как будто ее никогда не было. 
```

**операции с переменными**

```
1.VAR = v1 v2 v3   # заменяет список:     v1 v2 v3
2.VAR += v3 v4     # добавляет к списку:  v1 v2 v3 v3 v4
3.VAR -= v3        # убирает из списка:   v1 v2 v4
4.VAR *= v4 v5 v5  # добавляет, если нет: v1 v2 v4 v5
5.VAR += v5        # v1 v2 v4 v5 v5
6.VAR ~= s/v5/vvv  # заменяет по regexp:  v1 v2 v4 vvv v5
```

## Объектная нотация

**использование объектной нотации**

```
1.obj1.param1 = v1
2.obj1.param2 = v2
3.obj2.param1 = v3
4.obj2.param2 = v3
5.OBJLIST += obj1 obj2
6.# на самом деле нет никаких объектов, просто в имени переменной можно использовать точку
7.
8.s.f1 = v1
9.s.f2 = v2
10.s.f2.subfield = zzz
11.s = value
12.# это все - никак не связанные друг с другом переменные
```

**косвенное обращение к переменной**

```
1.s.f1 = 10
2.name = s
3.v = $${$${name}.f1}       # ошибка, так нельзя
4.eval(v = \$\$$${name}.f1)  # а вот так - можно
```

Функция eval вначале раскрывает переданный параметр согласно синтаксису .pro файла (получится v = \$$s.f1), а затем интерпретирует получившийся текст как часть .pro файла (в v запишется 10).

## Ветвления и циклы

### ветвления

```
1.scope { # скобка обязательно в этой же строке!
2.  ...
3.}
```

То что внутри поизойдет если scope истина.

Scope может быть

* `mkspecs`: названия доступных конфигураций qmake по сочетаниям платформа-компилятор, выглядят как `win32-msvc2008, macx-g++42` или `wince60standard-armv4i-msvc2005`. Могут указываться в виде grep масок (? и \*).
* Платформа: `win32, unix, macx`. Она косвенно определяется через mkspecs.
* Значения из специальной переменной `CONFIG` (которые являются именами фич, они же features). Для них тоже можно использовать маски, хотя практической пользы от этого немного.
* Встроенные условные функции.
* Пользовательские условные функции.
* build\_pass.

**Примеры**

```
1.win32-msvc* {
2.  # что-то специфическое для Visual C++ 2005-2012 под win32
3.  LIBS += advapi32.lib shell32.lib
4.  win32-msvc2012 {
5.     # в добавок что-то сугубо для 2012
6.  }
7.}
8.
9.*msvc2010 {
10.  # что-то для Visual C++ 2010 под любые платформы
11.  # вряд ли практический смысл в этом есть
12.}
13.
14.macx { 
15.  # что-то OSX специфическое
16.}
17.
18.CONFIG += console
19.console { # что-то только для консольных приложений
20.  DEFINES += _CONSOLE_
21.}
22.
23.exists(version.h) { # встроенная функция проверки 
24.                    # существования файла
25.  HEADERS += version.h
26.}
```

Если условие нужно только на одну строку, то можно записать scope в сокращенном варианте, через двоеточие. Например:

`console: DEFINES += _CONSOLE_`

**Вложенные условия так же можно сокращать (фактически : это логическое И)**

```
1.win32 {
2.  console {
3.     DEFINES += _CONSOLE_
4.  }
5.}
6.
7.# можно записать как
8.win32:console {
9.     DEFINES += _CONSOLE_
10.}
11.
12.# и даже как
13.win32:console: DEFINES += _CONSOLE_
```

**Логическое ИЛИ**

```
1.win32-msvc2005|win32-msvc2008 {
2.  # что-то только для Visual C++ 2005 или 2008
3.}
4.
5.!exists(version.h) {
6.  error(version.h is not found) # qmake завершиться с ошибкой
7.}
```

**Конструкция if позволяет объединить И и ИЛИ**

```
1.# debug режим на linux или OSX
2.if(linux-g++*|macx-g++*):CONFIG(debug, debug|release) {
3.    message("We are on Linux or Mac OS, and we are in debug mode.")
4.}
```

**else**

```
1.win32:xml {
2.    # xml под win32
3.    SOURCES += xmlhandler_win.cpp
4.} else:xml {
5.    # xml под юниксы
6.    SOURCES += xmlhandler.cpp
7.} else {
8.   # а без xml низзя
9.   error("Unknown configuration")
10.}
```

**for**

```
1.# Цикл выполняется для каждого значения из listVar
2.for(valueVar, listVar) { # скобка обязательно в этой же строке!
3.
4.   ...
5.}
```

```
1.EXTRAS = handlers tests docs
2.for(dir, EXTRAS) {
3.    exists($$dir) { 
4.        SUBDIRS += $$dir
5.    }
6.}
```

вторым параметром в цикле for может быть только имя переменной

```
1.EXTRAS = handlers tests docs
2.# $$unique убирает дубликаты
3.for(dir, $$unique(EXTRAS)) { # бананас
4.    exists($$dir) { 
5.        SUBDIRS += $$dir
6.    }
7.}
8.
9.# Это досадное ограничение можно обойти с помощью
10.# встроенной функции list. Она принимает одно значение и 
11.# создает переменную с уникальным именем, содержащим это значение.
12.EXTRAS = handlers tests docs
13.# $$unique убирает дубликаты
14.for(dir, $$list($$unique(EXTRAS))) { # работает!
15.    exists($$dir) { 
16.        SUBDIRS += $$dir
17.    }
18.}
```

**Существуют аналоги сишных операторов continue и break, это функции next() и break():**

```
1.LIST = 1 2 3
2.for(a, LIST) {
3.  equals(a, 1): next(); 
4.  equals(a, 3): break();
5.  message($$a)
6.}
7.# вывод цикла будет следующим:
8.# Project MESSAGE: 2
```

**for как условная функция**

```
1.LIST = 1 2 3
2.for(a, LIST):equals(a, 2):message($$a)
```

## Функции пользователя

**объявление функций**

```
1.# Условная функция используется в конструкциях scopes
2.defineTest(testFunctionName) {  # скобка обязательно в этой строке!
3.  ...
4.}
5.
6.# Обычная функция
7.defineReplace(replaceFunctionName) {  # скобка обязательно в этой строке!
8.  ...
9.}
```

**пример**

```
1.defineReplace(argsTest){
2.    message($$ARGS)  # вывод на консоль
3.    return($$2)      # возвращаем значение второго параметра
4.}
5.
6.# если возвращаемое значение не нужно, присваивание можно опустить
7.$$argsTest(1,2,3) # 1 2 3
8.VAR = $$argsTest(1,2 3) # 1 2 3
9.message($$VAR) # 2 3
```

```
1.# возвращает true, если все файлы, имена которых
2.# переданы как параметры, существуют
3.defineTest(allFiles) {
4.    files = $$ARGS
5.    for(file, files) {
6.        !exists($$file) {
7.            return(false)
8.        }
9.    }
10.    return(true)
11.}
12.
13.# пример использования
14.!allFiles($$SOURCES) {
15.  error("Not all required *.cpp files are found!")
16.}
```

* условные функции возвращают значения return(true) или return(false)
* функции обязательно должны что-то вернуть
* Условные функции вызываются в условиях конструкций ветвления с помощью привычного синтаксиса: `testFunction(param1, param2, ...)`. Скобки обязательны даже в том случае, если функция вызывается без параметров.
* Обычные функции вызываются аналогично, но зачем-то с использованием оператора $$: `$$replaceFunction(param1, param2, ...)\`
* в любую функцию можно передовать любое количество параметров
* обращаться к параметрам можно через порядковый номер `$$1``$$2`
* можно обратиться ко всем параметрам одновременно `$$ARGS`, что даст их значения, соединенные проблемы. (значение параметра не должно содержать пробелы)

**определена ли функция**

```
1.defineTest(testtest) {
2.  return(false)
3.}
4.
5.# поиск условной (test) функции с именем testtest
6.defined(testtest, test) {
7.  message(There is \'testtest\' test function)
8.}
9.
10.# поиск обычной функции (replace) с именем testtest
11.!defined(testtest, replace) {
12.  message(There is no \'testtest\' replace function)
13.}
14.
15.# Можно и без типа искать
16.!defined(testtesttest) {
17.  message(There is no function of any type with name \'testtesttest\')
18.}
19.
20.# Можно искать также и переменные
21.defined(SOME_VARIABLE, var) {
22.  message(SOME_VARIABLE does exist.)
23.}
```

* все функции работают с локальными копиями переменных, создаваемых на момент вызова. Если изменить переменную в функции, то это изменение никак не отразится на этой переменной в месте вызова
* создание новой переменной в функции не создаст переменной в месте вызова
* чтобы перенести изменения в контекст вызова, необходимо использовать встроенную функцию `export(ИмяПеременной)`. Переменная переносится во все контексты, вплоть до файла.

```
1.defineReplace(func1) {
2.  AAA = newValue
3.  export(AAA)
4.  return(0)
5.}
6.
7.defineReplace(func2) {
8.  $$func1()
9.  message(func2: $$AAA)
10.  return(0)
11.}
12.
13.$$func2()               # func2: newValue
14.message(global: $$AAA)  # global: newValue
```

## include модульность

```
1.# можно вынести общие настройки для группы проектов,
2.# например свои функции, в один общий файл
3.include(../common.pri)
4.
5.# можно разбить проект на части, и каждую
6.# часть вести в своем каталоге
7.include(subsystem1/subsystem1.pri)
8.include(subsystem2/subsystem2.pri)
9.
10.# можно прописать настройки по подключению своих
11.# статических библиотек в специальном месте,
12.# и подключать их к проекту одной строкой
13.include(d:/projects/libs/my_library.pri)
```

* путь к включаемому файлу абсолютный или относительный(отнсительно файла)
* существеут встроенная переменная `_PRO_FILE_PWD_`, которая содержит путь к каталогу самого первого файла, который начал обрабатывать qmake.

```
1.# part/subpart/subsubpart/subsubpart.pri
2.# вместо include(../../../common.pri)
3.include($${_PRO_FILE_PWD_}/common.pri)
```

* если включаемый файл не найден, будет выдано предупреждение
* такие ситуации можно обрабатывать в условиях

```
1.!include(lib1.pri) {
2.  # обработка ошибки, например включение заменяющего файла
3.  include(lib2.pri)
4.}
```

модульность можно создать при помощи файлов .pri, просто включив их основной .pro `include(module1/module1.pri) include(module2/module2.pri)`

## Дефолтные значения переменных

QMAKE\_SPEC - содержат дефолтный mkspec (различные тулчейны для разных платформ)

QMAKE\_XSPEC - то же

`QMAKESPEC` - полный путь к каталогу дефолтного (или своего) mkspec

`.qmake.cache`- файл, который неявно включается во все .pro файлы ниже своего уровня, для хеширования???

Значения переменных через командную строку qmake

```
1.qmake "CONFIG+=test" "VAR1 = VAR1" VAR2=VAR2 "VAR3=with spaces" test.pro
```

Переменные окружения

```
1.# замена $$(ENV_VAR) на соответствующее значение произойдет
2.# во время выполнения qmake, не make
3.VAR = $$(ENV_VAR)
```
